## 一、研发模式：三端实践全景

### 1.1 研发模式选型框架

字节内部沉淀了三种核心研发模式，适用于不同协作复杂度和交付频率：

| 研发模式 | 特点 | 协作复杂度 | 推荐分支模型 |
|---------|------|-----------|------------|
| **需求研发流程** | 围绕单需求组织交付，无固定时间窗口 | 中 | 主干集成、主干发布 |
| **火车研发流程** | 多需求并行开发、集成测试、集中交付 | 高 | 分支集成、主干/分支发布 |
| **主干开发(TBD)** | Small MR持续合入主干，feature gating控制可见性 | 工程难度高但流程简单 | 主干开发、good commit发布 |

**选型决策因素**：团队规模、发布频率、自动化测试完备度、需求并行度。

### 1.2 分支模型

四种分支模型，复杂度递增：

1. **主干集成 + 主干发布**：最简单，适合串行低频场景。弊端：回归测试占用主干。
2. **分支集成 + 主干发布（推荐）**：火车模式，回归测试在分支上进行不占主干，撤销代码方便。
3. **分支集成 + 分支发布**：多个并行需求互不干扰，但归档时冲突解决有风险。
4. **主干开发 Trunk-based**：Monorepo几乎只能用此模型，需要丰富的自动化测试和feature gating机制。

### 1.3 服务端研发实践七要素

服务端研发模式包含七个工程活动实践：

1. **研发工作流**：定义开发→测试→发布核心流程及各角色职责
2. **需求交付**：工程交付流与需求交付流的联动（单需求/多需求集中交付）
3. **分支模型**：分支创建、合入、同步、归档、销毁规则
4. **环境管理**：BOE泳道→PPE泳道的分层验证，动态创建独立泳道保隔离性
5. **质量门禁**：测试准入、测试准出、发布准入的固定卡点槽位
6. **部署策略**：按阶段/按集群/蓝绿/分批次等多种升级模式
7. **变更观测**：发布过程的告警、监控观测体系

**关键实践原则**：
- 常规流程和Hotfix流程分离，Hotfix具有优先发布权
- 环境建议：每次测试任务动态创建独立泳道，发布后自动回收
- BOE基准环境应尽可能与线上一致，成功发布后自动同步

### 1.4 客户端多仓合码流程

客户端研发的核心特点：**强版本发布 + 大规模协同 + 高并发需求 + 代码复用**。

**多仓架构**：
- **主仓**：壳工程，编译构建最终产物
- **子仓**：以二进制产物形式集成到主仓，中台业务以子仓形式服务多个应用

**合码流程关键环节**：
1. 组件预发布 → CI Check（主仓+子仓流水线、冲突检查、安全合规）
2. 人工评审（四级：研发任务级/应用级/仓库级/文件级）
3. 人工测试（自测→提测→测试验证）
4. **合入锁**：确保合码时序一致性，避免并发冲突
5. 组件发布和子仓合入 → 主仓集成 → 主仓合入

**封版流程**（解决版本需求完整性问题）：
- 设定版本截止时间，提前2-4小时开始封版
- 对develop分支加锁，需求准入管控（机器人催促+上下车机制）
- 所有可合入需求完成后，检出release分支，释放develop锁

**TikTok案例收益**：版本协作等待时间 6小时→0小时，灰度人力 8人时/轮→2人时/轮。

### 1.5 前端研发SDLC全生命周期

前端研发覆盖7个阶段，每个阶段都有对应的工具和最佳实践：

| 阶段 | 关键能力 | 推荐工具 |
|------|---------|---------|
| 项目创建 | 脚手架、一键创建 | EdenX, Cloud-CLI |
| 编码调试 | 智能补全、云研发 | Codeverse, FESuite |
| 构建 | 依赖缓存、CDN加速 | SCM缓存、全球CDN |
| 自动化测试 | 单元/E2E/页面分析 | PagePass, Perfsee |
| 集成 | 代码分析、CR、CI | Codebase CI |
| 部署 | 灰度、容灾兜底 | Goofy Deploy, Gecko |
| 监控报警 | 报警分级、值班 | Slardar Web |

**前端三种发布模式**：
- **快速发布模式**：无需开发任务，直接发布单，适用于紧急上线
- **需求交付模式**：1个发布单关联1个开发任务，适用于快速迭代
- **火车发布模式**：1个发布单关联多个开发任务，适用于版本交付

### 1.6 Bits平台升级（从ByteCycle 1.0到Bits 2.0）

**核心升级价值**：
- **统一产品模型**：用"开发任务+发布单"两个实体统一承载所有研发模式（ByteCycle曾有7种产品形态）
- **流水线极大简化**：核心功能产品化（分支管理、Code Review、准入准出等），不再依赖流水线原子服务
- **强约束核心环节**：质量管控从自由配置变为固定卡点槽位
- **度量口径统一**：标准化流程定义使数据化管理成为可能

## 二、代码质量保障体系

### 2.1 单元测试

#### 核心理念

**测试金字塔**：70%单元测试 + 20%集成测试 + 10%E2E测试（Google建议）。当前多数业务呈"蛋筒冰激凌"反模式（手工测试最多，单测最少）。

**单元测试的五大价值**：
1. 提高代码质量：快速反馈，早发现早修复
2. 提高研发效率：无需手动测试，缺陷定位快
3. 简化重构："安全网"效应
4. 改善设计：迫使以更小粒度思考代码
5. 文档功能：说明代码预期行为

#### 三阶段实践路径

**第一阶段（1-3月）：会写、会用**
- 培养意识和习惯，从新需求代码开始
- 集成CI流水线（关键！）
- 初期不设覆盖率卡口或仅设低阈值（如增量30%）
- 在排期中加入单测排期

**第二阶段（3-12月）：写好、有效**
- 要求高质量单测（有断言、断言充分、覆盖不仅Happy Path）
- 为核心模块存量代码补充单测
- 正式开启覆盖率卡口：**增量行覆盖率 > 80%，全量行覆盖率 > 60%**

**第三阶段（理想状态）**：提升可测性，尝试TDD

#### 关键原则
> **高质量发展，不要"先污染后治理"**。不建议为快速提高覆盖率而批量生成低质量单测。严重污染的结果通常是放弃而非治理。

**微软VSTS案例**：花两年半从大型系统测试为主转变为单元测试为主，缺陷趋势显著下降。

### 2.2 代码评审

#### 核心价值

- Google: 80%代码评审带来代码改善
- 未审查代码引入缺陷概率是已审查的2倍
- 代码评审使开发者了解的独特文件增加66%-150%

#### 量化指标体系

**北极星指标**：
| 指标 | 说明 |
|------|------|
| 合入MR评审率 | 达成评审条件的MR/合入MR总数 |
| 合入MR有效评审率 | 1-闪电评审率（阅读速度>200行/分钟视为闪电评审） |
| 合入MR评论率 | 含非author创建Thread的MR/MR总数 |
| 评审问题发现数 | Outdated Thread数量（评论后代码被修改） |
| 评论解决率 | Resolved & won't fix thread / thread总数 |

#### 实践成熟度三阶段

| 阶段 | 指标特征 | 改进措施 |
|------|---------|---------|
| 低实践 | 评审率低 | 建立评审意识、配置评审规则 |
| 形式化 | 评审率高但闪电评审率高 | 排期预留CR时间、控制变更大小、选合适Reviewer |
| 效率提升 | 评审率高、有效率高但周期长 | 分析各阶段时长，针对性优化 |

#### 代码所有权（OWNERS文件）

通过声明开发者与目标对象关系，明确维护代码段的权利和义务：
- 提升变更效率：熟悉代码评审更快
- 提升代码质量：保持风格和设计一致性
- 降低协作成本：更容易找到合适沟通对象

#### 评审交流原则

**第一原则**：每个人都是为了代码库中的代码质量变好。

- 没有"完美"代码，只有更好的代码
- 评审者不应追求完美而应追求持续改进
- 非重要评论加"Nit:"前缀
- 三种批准：Approve / Approve later changes / Disapprove

### 2.3 静态代码分析

#### 现状与价值

**字节现状数据**：
- MR阶段问题拦截率29.70%，严重问题拦截率38.14%
- 代码规范拦截率35.69%，强制规范拦截率74.68%
- 存量问题修复率仅9.71%
- 超高复杂度函数（圈复杂度>15）超过361,927个

**核心价值**：代码质量提升和需求交付不冲突，放任代码质量恶化会导致负重前行。

**Google参考**：截至2018年1月，C++和Java编译器错误均清零，所有问题在代码审查前解决。

#### 实践目标

- 增量代码问题拦截率 > 40%，严重问题 > 50%
- 代码规范拦截率 > 50%，强制规范 > 80%
- 高复杂度函数逃逸率 < 30%
- 存量问题修复率 > 20%

#### 五步实践流程

1. **制定代码质量提升目标**：评估现状，明确提升方向和规则
2. **定义衡量指标**：关键指标+MR门禁要求+计划
3. **接入代码分析平台**：配置规则集→开启全量离线分析→本地IDE→集成MR/流水线
4. **持续消费扫描结果**：增量（卡点要求消费）+ 存量（高优问题排期修复）
5. **定期更新质量标准**：建立规则/规范/门禁的定期更新机制

#### 门禁配置建议

**增量代码门禁**：
- Critical & Error等级问题 > 0（平台默认）
- 高复杂度函数数量限制
- 代码重复度限制

**存量治理优先级**：
- 圈复杂度 > 21的函数优先处理
- 认知复杂度 > 15的函数优先处理
- 重复度 > 30%的文件优先处理

#### 架构前端案例

MR阶段问题拦截率从14.21%提升至**57.46%**，严重问题拦截率从14.66%提升至**75.92%**，强制规范拦截率达**94.03%**。

## 三、监控报警体系

### 3.1 监控四层模型

从最接近用户到系统底层：

| 层次 | 说明 | 字节内承载平台 |
|------|------|-------------|
| **RUM** | 真实用户体验（APP/Web端性能异常） | Slardar |
| **业务监控** | 业务场景度量指标（如下单成功率） | Argos（建设中） |
| **应用监控** | 微服务及依赖组件监控 | Argos |
| **基础设施监控** | 网络、服务器主机 | SYS各平台 |

### 3.2 服务端监控报警

#### 数据采集三支柱

| 数据类型 | 说明 | 采集组件 |
|---------|------|---------|
| **Metrics** | 时序数据（QPS、CPU利用率） | ByteTSD/Metrics 2.0 |
| **Logging** | 结构化/非结构化日志 | Streamlog |
| **Tracing** | 微服务调用链路 | BytedTrace |

**推荐**：服务性能指标接入BytedTrace（自动生成APM看板+自动注入报警规则），其他指标接入Metrics 2.0。

#### 报警分级规范

| 级别 | 适用场景 | 通知渠道 | 通知策略 |
|------|---------|---------|---------|
| **Critical** | 业务可用性指标（如下单成功率） | Lark+短信+电话 | 即时通知，不降频，有升级机制 |
| **Warning** | 可能引发稳定性问题（如QPS波动） | Lark+短信+电话 | 即时通知，支持降频 |
| **Notice** | 内因指标（如CPU利用率） | Lark | 仅1h/24h聚合报表 |

**降噪策略**：
- Warning级别开启通知降频
- 开启PSM聚合通知（按周期聚合低级别报警）
- 低级别报警不配置电话通知

#### 报警升级机制

三级升级，Critical级别无人响应自动触发：
- **一级升级**：通知方式升级为电话，可选升级到指定值班计划
- **二级升级**：升级到Leader / Owner / 祖先节点值班人
- **三级升级**：继续沿服务树向上升级

#### 智能报警

基于无监督机器学习算法（TSAP平台），从历史时序数据提取特征：
- 适用于：周期规律型（QPS）、平稳型（错误率）、无规律波动型（Latency）
- 优点：配置简单，自适应不同子序列阈值，适应周期性变化

#### 自动归因

单实例故障的自动诊断与修复：容器化部署环境下，通过自动迁移解决单实例问题（系统资源、运行时、业务逻辑三类原因）。

### 3.3 前端&客户端监控报警

#### 现状

- Slardar日均报警事件4万+，仅**4%**被响应
- 部分业务报警召回率仅30%
- 近半年事故发现时长PCT80为40+小时

#### 报警生命周期SOP

六个环节形成闭环：
1. 报警规范制定 → 2. 监控场景梳理 → 3. 报警规则配置/注入 → 4. 报警响应 → 5. 报警跟进处理 → 6. 阶段性复盘

#### 指标分类与策略建议

| 指标类型 | 推荐策略 | 关键配置 |
|---------|---------|---------|
| **稳定性（异常率）** | 阈值报警+梯度策略（120%/150%/200%） | 必须配置样本量阈值，P0频率5分钟 |
| **稳定性（异常数）** | 同环比波动策略 | 同比1天+3天+7天组合判断 |
| **新增错误** | 新增错误报警/订阅 | 关注新上线代码引入的新错误类型 |
| **流量类（PV/UV）** | 同环比或智能报警 | 周期性数据适合智能报警 |
| **性能类** | 阈值报警（PCT75/PCT90） | 非P0建议小时/天级频率，搭配灰度监控 |
| **自定义事件** | 阈值报警 | 成功率类指标样本量阈值选"分母" |

#### 报警等级设定

- **P0**：需立即修复（热修/回滚），影响核心功能 → Lark+电话，开启升级策略
- **P1**：可跟版修复，影响部分用户非核心功能 → Lark+短信（可选）
- **P2**：长期优化，仅影响体验 → 仅Lark群，开启降频

#### 响应处理基本原则

**1-5-10原则**：1分钟发现 → 5分钟定位 → 10分钟恢复

**排障思路**：
1. 首先检查变更事件（70%事故与变更有关）
2. crash/JS错误：堆栈聚类分析 → sourcemap反解 → session复现
3. 请求错误：错误码分布 → logid关联后端日志
4. 静态资源错误：资源类型/名称归因 → CDN排查
5. 性能指标：智能归因 + Perfsee工具

#### 治理建议

| 问题 | 治理手段 |
|------|---------|
| 报警数过大 | 按量排序调阈值，非严重降级为P2，检测频率调大 |
| 响应率低 | 及时打标，定期批量处理，自动巡检 |
| 有效率低 | 调整时间窗口，使用PCT报警过滤尖刺 |

## 五、通用方法论提炼

### 5.1 质量与效率的平衡

字节的实践揭示了一个核心张力：质量管控越严格，效率损失越大。解法是：
- **产品化 > 流水线化**：将核心能力从流水线原子服务升级为产品功能（Bits 2.0的核心理念）
- **半固化框架**：提供开箱即用的标准流程，业务只需微调
- **自动化 > 人工**：智能报警、自动归因、精准测试替代人工

### 5.2 分层治理模型

无论是监控（RUM/业务/应用/基础设施）、代码质量（规范/可靠性/可维护性）、还是测试（单元/集成/E2E），都遵循**分层治理**原则：
- 每层有明确的责任边界
- 从底层到顶层，覆盖面递减但业务相关性递增
- 自动化能力集中在底层，人工判断集中在顶层

### 5.3 渐进式推进策略

所有实践都遵循渐进推进：
1. **单测**：会写会用(1-3月) → 写好有效(3-12月) → TDD
2. **代码分析**：离线摸底 → 小范围MR开启 → 全量+门禁
3. **监控报警**：覆盖率优先 → 降噪治理 → 智能化
4. **代码评审**：低实践 → 形式化 → 效率提升

### 5.4 度量驱动改进

每个实践都定义了明确的量化指标和目标：
- **过程指标**（如拦截率、评审率、响应率）驱动行为改变
- **结果指标**（如缺陷数、事故数、MTTR）验证收益
- **定期复盘**形成闭环
