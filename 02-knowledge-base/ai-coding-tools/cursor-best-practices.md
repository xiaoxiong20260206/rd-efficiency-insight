## 🎯 三种模式概览

### Ask 模式 - 即时对话助手
**特点**：
- 实时交互式编程助手
- 适合快速问答和小范围代码修改
- 上下文感知当前编辑的文件
- 支持多轮对话保持上下文

**适用场景**：
- 代码解释与学习
- Bug调试
- 技术方案对比
- 快速澄清技术细节

**最佳实践**：
- 提供具体上下文，选中相关代码段
- 使用渐进式提问，从简单到复杂
- 善用 `@文件名` 引用特定文件
- 利用 `@docs` 引用文档上下文

### Agent 模式 - 自主编程代理
**特点**：
- 自主探索、规划和执行复杂代码库变更
- 可以独立完成多文件修改
- 具备项目全局视图和架构理解
- 支持复杂任务的分解和执行

**适用场景**：
- 功能模块开发
- 代码重构优化
- 多文件协调修改
- 自动化任务执行

**最佳实践**：
- 给出清晰的需求边界和验收标准
- 提供足够的上下文和约束条件
- 定期检查进度并提供反馈
- 使用 `/pause` 命令在关键节点暂停

### Background 模式 - 后台异步代理
**特点**：
- 异步远程代理，可作为24/7 AI工作队伍
- 支持Slack集成，理解完整线程上下文
- 适合长时间运行的任务
- 数据保留几天用于上下文连续性

**适用场景**：
- CI/CD流水线优化分析
- 代码质量全面审查
- 大型项目架构设计
- 长时间数据处理

**最佳实践**：
- 适合耗时的分析和报告任务
- 设置明确的任务边界和输出格式
- 利用Slack集成进行团队协作
- 定期清理不再需要的后台任务

## 📝 Cursor Tab 功能最佳实践

### 一、上下文优化策略

#### 1.1 保持相关文件打开
- **原理**：Tab会分析所有打开的文件来理解项目结构
- **实践**：保持3-5个核心文件在标签页中

#### 1.2 利用最近编辑历史
- **原理**：Tab优先参考最近编辑的代码模式
- **实践**：先写一个完整示例，后续相似函数获得更准确补全

### 二、注释引导技巧

#### 2.1 详细的函数注释
```javascript
/**
 * 验证用户输入的邮箱格式
 * @param {string} email - 用户输入的邮箱
 * @returns {boolean} 是否为有效邮箱格式
 */
function validateEmail(email) {
  // Tab 会根据注释生成符合描述的代码
}
```

#### 2.2 步骤式注释
```javascript
async function processOrder(order) {
  // 1. 验证订单数据
  // 2. 检查库存
  // 3. 计算价格
  // 4. 创建支付请求
  // Tab 会根据步骤生成结构化代码
}
```

### 三、命名规范与模式识别

#### 一致的命名约定
- 使用语义化命名：`getUserById`, `updateUserProfile`
- 保持前缀一致：`handle-`, `fetch-`, `validate-`
- Tab能识别常见设计模式（如Builder模式）

### 四、高效工作流

#### 渐进式编码
1. 先写函数签名
2. 添加注释说明
3. 写第一行关键逻辑
4. 让Tab补全剩余部分

#### 快捷键组合
| 快捷键 | 功能 |
|--------|------|
| `Tab` | 接受补全 |
| `Cmd/Ctrl + →` | 接受部分补全（逐词） |
| `Escape` | 拒绝补全 |
| `Alt + \` | 手动触发补全 |

### 五、性能优化

#### 减少干扰因素
- 关闭无关的大文件
- 清理临时文件和构建产物
- 保持代码格式一致

#### 索引优化
- 定期重启Cursor刷新索引
- 使用 `.cursorignore` 排除不需要的目录

### 六、常见问题解决

| 问题 | 解决方案 |
|------|----------|
| 补全速度慢 | 减少打开文件数量、检查网络、清理缓存 |
| 不符合项目风格 | 添加 `.cursorrules` 文件指定编码规范 |
| 补全内容重复 | 避免相似代码块、使用更具体的命名 |

## 💡 核心原则

1. **上下文为王** - 提供充分的上下文信息
2. **模式驱动** - 建立并保持一致的编码模式
3. **循序渐进** - 从简单到复杂，让AI学习你的风格
4. **持续优化** - 根据实际效果调整使用策略
