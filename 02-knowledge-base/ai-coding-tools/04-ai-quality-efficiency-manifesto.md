## 📜 AI质效宣言

> 发布时间：2025年10月 QECon大会北京站
> 发起单位：QECon组委会

### 核心理念

我们致力于在保持高质量标准的同时，系统性提升研发效能。在此过程中形成了如下核心价值观，着眼于软件工程的长期健康发展，通过价值锚定，让"质效平衡"成为行业共同遵循的底层逻辑。

### 六大核心原则

| 左项（更重视） | 对比 | 右项（有价值） |
|---------------|:----:|---------------|
| **协作共生** | > | 局部取代 |
| **长效质量** | > | 短期提效 |
| **适应进化** | > | 固化流程 |
| **价值创造** | > | 技术崇拜 |
| **数据驱动** | > | 经验依赖 |
| **开放共享** | > | 封闭自建 |

> **"尽管右项有其价值，我们更重视左项的价值。"**

### 核心要点解读

| 原则 | 含义 |
|------|------|
| **协作共生 > 局部取代** | AI与人的协作共生，而非简单取代 |
| **长效质量 > 短期提效** | 关注长期质量而非短期效能提升 |
| **适应进化 > 固化流程** | 保持适应性进化，避免流程僵化 |
| **价值创造 > 技术崇拜** | 以业务价值创造为核心，避免盲目追求技术 |
| **数据驱动 > 经验依赖** | 用数据驱动决策，减少主观经验依赖 |
| **开放共享 > 封闭自建** | 倡导开放共享，而非封闭自建 |

## 🎯 智能化编码工具分层竞争格局（L1-L5）

> 原文链接：https://prompt.16x.engineer/blog/ai-coding-l1-l5
> 发布时间：2025年7月

借用自动驾驶汽车的术语，将智能化编码工具分为五个级别：L1 至 L5。

### 📊 五级分层总览

```
L1 (Code Completion)     → 代码补全
        ↓
L2 (Task Automation)     → 任务级自动化
        ↓
L3 (Project Automation)  → 项目级自动化
        ↓
L4 (AI Software Engineer) → AI软件工程师
        ↓
L5 (AI Development Teams) → AI开发团队
```

---

### 🔹 L1 - Code Completion（代码补全）

**定义**：最基础的层面，专注于代码补全

**特点**：
- 通过智能建议和自动补全简化编码过程
- 已成为现代开发环境中的常态
- 为更先进的AI编码工具铺平道路

**代表产品**：
- GitHub Copilot
- TabNine
- Codeium

---

### 🔹 L2 - Task-Level Automation（任务级自动化）

**定义**：根据描述性提示处理开发任务

**特点**：
- 能够处理新功能开发、错误修复、代码重构等任务
- 需要精心设计的提示以及相关的源代码上下文
- 提供更友好的用户界面与LLM交互

**挑战**：
- 使用大型语言模型进行编码任务可能颇具挑战性
- 手动创建高质量提示耗费时间且枯燥

**产品分类**：

| 类别 | 产品 |
|------|------|
| **LLMs** | ChatGPT, Claude |
| **工作流自动化工具** | aider (CLI), 16x Prompt (桌面应用) |
| **IDE扩展** | Cline, Roo Code, Continue, Amazon Q Developer |
| **AI原生IDE** | Replit CDE, Cursor, PearAI, Windsurf, Trae |

---

### 🔹 L3 - Project-Level Automation（项目级自动化）

**定义**：项目级自动化的早期阶段

**特点**：
- 能够分析项目、理解需求、生成拉取请求
- 自动化软件开发流程中的多个步骤
- 与项目管理工具（Jira）和代码平台（GitHub）集成

**局限性**：
- 仍处于初级阶段
- 仅能管理相对简单的编码项目
- 需要人工干预确保代码质量
- 通常只处理技术栈的一部分（如前端）

**产品分类**：

| 类别 | 产品 |
|------|------|
| **一线厂商工具** | Claude Code (Anthropic), Codex (OpenAI), Jules (Google) |
| **付费产品** | Codegen, Sweep, v0 (Vercel), Tempo |
| **免费产品** | CerebrasCoder, LlamaCoder, AppCrapper |
| **部分开源** | Pythagora, Plandex |

---

### 🔹 L4 - AI Software Engineer（AI软件工程师）

**定义**：从人工驱动到AI驱动软件开发的关键转变

**特点**：
- 从产品需求到生产部署的全面自动化
- 能够使用终端和部署工具
- 让非技术人员能在几分钟内打造功能完备的软件产品
- 解读产品需求、管理代码部署、维护生产环境

**产品分类**：

| 类别 | 产品 | 备注 |
|------|------|------|
| **全栈Web开发** | Replit Agent, Marblism, bolt.new, Trickle, Lovable | - |
| **公开可用** | devlo | 54.20 on SWE-bench Verified |
| **公开可用** | Gru | 45.20 on SWE-bench Verified |
| **公开可用** | Devin | $500/250 ACUs |
| **联系销售** | Engine | 51.80 on SWE-bench Verified |
| **联系销售** | Factory Code Droid | 19.27 on SWE-bench Full |
| **等候名单** | Solver, Genie, Tessl | - |

---

### 🔹 L5 - AI Development Teams（AI开发团队）

**定义**：由多个AI软件工程师组成的协作系统

**特点**：
- 多个AI Agent可以协作完成一个项目
- 每个Agent专注于软件开发的不同方面
- 实现真正的多智能体协同

**理论基础**：
> 微软《AutoDev》论文提出：Agent可以"从代理调度器接收目标和对话历史记录，并根据规则和操作配置做出响应"

**产品分类**：

| 类别 | 产品 |
|------|------|
| **研究** | AutoGen, AutoDev |
| **开源** | MetaGPT |
| **商业化等待名单** | MGX (MetaGPT团队) |
| **商业化联系销售** | MultiDevin |

**展望**：
- GPT-5预计于2025年发布
- AI开发团队的梦想已不再遥不可及
- AI将处理软件开发中的例行工作，使开发人员专注于更高层次的任务

## 🔮 未来编程范式：Vibe模式 vs Spec模式

> 发布时间：2025年8月6日

### 核心观点

AI编程产品正经历从辅助工具到自主代理的深刻变革。**Vibe模式与Spec模式作为两种核心范式，分别聚焦在创新探索与复杂工程落地的不同场景**，未来很可能形成互补共生的主流趋势。

### 两种模式对比

| 维度 | Vibe Coding（氛围编码） | Spec模式（规约模式） |
|------|------------------------|---------------------|
| **定位** | 创新探索、快速原型开发 | 复杂工程落地、规范化流程 |
| **特点** | 灵活、自由、创意驱动 | 结构化、规范化、可维护 |
| **适用场景** | 个人项目、原型验证、创意探索 | 企业内部大型复杂系统 |
| **代表工具** | Cursor, Windsurf | Kiro (AWS) |

### Spec模式详解

**代表产品**：AWS Kiro

**核心理念**：
> **"Go from vibe coding to viable code"**
> （从氛围编码走向可行代码）

**流程**：
```
用户提示 → 清晰需求 → 结构化设计 → 严格测试验证 → 高级Agent生成代码
```

**适用场景**：
- 企业内部研发大型复杂系统
- 需要符合架构约束确保长期可维护性
- 面向复杂场景的AI编码规范流程
- 融入成熟可靠的工程实践

**确保AI生成的代码是**：
1. ✅ 符合计划的
2. ✅ 可用的
3. ✅ 可维护的

### 范式选择建议

| 项目类型 | 推荐模式 |
|----------|----------|
| 个人side project | Vibe Coding |
| 原型快速验证 | Vibe Coding |
| 企业核心系统 | Spec模式 |
| 需要长期维护的项目 | Spec模式 |
| 创意探索阶段 | Vibe Coding → Spec模式 |

## 💡 核心洞察

### 2025年AI编码的关键趋势

1. **从辅助到自主**：AI编程工具正从"代码补全"向"自主完成任务"演进
2. **多智能体协同**：单一Agent向多Agent协作转变
3. **范式分化**：Vibe模式与Spec模式分别服务不同场景
4. **质效平衡**：AI质效宣言强调协作共生、长效质量

### 给开发者的建议

1. **拥抱变化**：AI编程是不可逆的趋势
2. **选择合适层级**：根据任务复杂度选择L1-L5工具
3. **掌握两种范式**：Vibe模式用于探索，Spec模式用于落地
4. **关注质效平衡**：遵循AI质效宣言的核心原则
