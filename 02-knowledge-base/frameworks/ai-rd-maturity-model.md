# AI研发成熟度模型（L1/L2/L3）

> 基于快手、字节、阿里、腾讯四大厂AI研发实践总结
> 
> 来源：《AI × 研发效能 2026洞察报告》
> 
> 整理时间：2026年3月9日

---

## 模型概述

AI研发成熟度模型定义了企业AI研发能力的三个递进层级，从个人辅助到组织协同，再到AI自主。

```
                    L3 AI自主
                   /  人如PM  \
                  /   AI端到端  \
                 ▲───────────────▲
                /    L2 AI协同    \
               /   人+AI深度协同   \
              ▲─────────────────────▲
             /      L1 AI辅助        \
            /     人主导，AI编码辅助    \
           ▲───────────────────────────▲
```

---

## L1：AI辅助（AI Assistance）

### 定义
人主导研发过程，AI仅在编码环节提供辅助支持。

### 核心特征

| 维度 | 特征 |
|------|------|
| **人机关系** | 人主导，AI辅助 |
| **AI参与范围** | 编码环节（代码补全、生成、解释） |
| **决策权** | 完全由人掌控 |
| **典型场景** | 代码补全、函数生成、注释生成、简单Bug修复 |

### 能力要求

#### 对AI工具的要求
- 代码补全准确率 > 70%
- 支持主流编程语言和框架
- 与IDE深度集成
- 响应速度 < 500ms

#### 对开发者的要求
- 理解AI生成代码的含义
- 具备代码审查能力
- 掌握Prompt工程基础

### 度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **代码生成采纳率** | > 30% | AI生成代码被开发者采纳的比例 |
| **代码补全使用率** | > 80% | 开发者使用AI补全的频率 |
| **人均编码效率提升** | 20-40% | 相比无AI辅助的编码效率提升 |

### 大厂实践数据

| 公司 | L1采纳率 | 产品 |
|------|---------|------|
| 快手 | ✓ 全面覆盖 | Kwaipilot |
| 阿里 | △ 27% | 通义灵码 |
| 字节 | △ ~20% | Trae（内部未完全打通） |
| 腾讯 | △ 部分BG | CodeBuddy |

### 实施建议

1. **工具选型**：选择与公司技术栈匹配度高的AI编程助手
2. **IDE集成**：确保与现有开发环境无缝集成
3. **培训推广**：开展Prompt工程培训，提升开发者AI协作能力
4. **度量基线**：建立L1采纳率的度量体系

---

## L2：AI协同（AI Collaboration）

### 定义
人和AI深度协同，AI参与研发全流程（需求→设计→编码→测试→发布）。

### 核心特征

| 维度 | 特征 |
|------|------|
| **人机关系** | 深度协作，分阶段主导 |
| **AI参与范围** | 全流程：需求理解、架构设计、编码、测试、Code Review |
| **决策权** | 人做关键决策，AI执行具体任务 |
| **典型场景** | 需求自动拆解、测试用例生成、CR辅助、文档自动生成 |

### 能力要求

#### 对AI工具的要求
- 理解业务上下文
- 支持全流程自动化
- 与现有研发平台深度集成
- 支持人机协作的交互模式

#### 对开发者的要求
- 从"执行者"转变为"协调者"
- 具备需求拆解和验证能力
- 掌握AI协作方法论

#### 对组织的要求
- 标准化研发流程
- 完善的需求管理体系
- 清晰的人机分工边界

### 度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **L2需求占比** | > 20% | 采用L2模式完成的需求比例 |
| **需求交付周期缩短** | 30-40% | 相比传统模式的交付周期 |
| **自动化测试覆盖率** | > 60% | AI生成测试用例的覆盖度 |
| **CR效率提升** | 50%+ | Code Review效率提升 |

### 快手最佳实践

#### 成果数据

| 指标 | 数据 |
|------|------|
| L2&L3需求占比 | **20.34%** |
| 标杆团队交付周期下降 | **58%** |
| 主站技术部人均交付需求增长 | **80%+** |

#### 关键实践

1. **效能中心主导**：由效能团队而非AI团队主导L2落地
2. **组织效能视角**：以需求交付周期为核心度量指标
3. **平台深度集成**：Kwaipilot与KDev平台深度集成
4. **标杆团队试点**：选择2-3个团队先行试点

#### L2开发方法

```
需求理解 → AI辅助拆解 → 人确认 → AI生成代码 → 人审查 → AI生成测试 → 人验证 → 发布
   ↑________________________________________________↓
                        循环迭代
```

### 实施建议

1. **选择试点团队**：选择流程规范、技术栈统一的团队
2. **定义L2需求标准**：明确什么样的需求适合L2模式
3. **建设平台能力**：确保AI工具与研发平台深度集成
4. **建立反馈机制**：持续收集开发者和AI的协作反馈
5. **培训推广**：总结最佳实践，在全公司推广

## L3：AI自主（AI Autonomy）

### 定义
人类似产品经理，把需求交给AI端到端完成，人主要负责验收和方向把控。

### 核心特征

| 维度 | 特征 |
|------|------|
| **人机关系** | 人如PM，AI如开发团队 |
| **AI参与范围** | 端到端：需求→代码→测试→部署→监控 |
| **决策权** | AI自主决策，人做最终验收 |
| **典型场景** | 标准化需求自动完成、低复杂度功能开发、Bug自动修复 |

### 能力要求

#### 对AI工具的要求
- 具备端到端交付能力
- 自主决策和问题解决能力
- 完善的监控和回滚机制
- 支持人机协作的接管机制

#### 对开发者的要求
- 从"开发者"转变为"产品经理+架构师"
- 具备需求定义和验收能力
- 掌握AI能力边界，知道何时人工介入

#### 对组织的要求
- 高度标准化的研发流程
- 完善的质量保障体系
- 清晰的AI能力边界定义

### 度量指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| **L3需求占比** | > 10% | 采用L3模式完成的需求比例 |
| **需求交付周期** | < 1天 | 从需求提交到上线的周期 |
| **AI交付成功率** | > 90% | AI端到端交付的成功率 |
| **人工介入率** | < 20% | 需要人工介入修正的比例 |

### 当前状态

L3目前仍处于**探索阶段**，各厂都在小规模试点：

| 公司 | L3状态 |
|------|--------|
| 快手 | 试点探索中 |
| 阿里 | 反思调整中，尚未规模化 |
| 字节 | 部分探索，未解决效率问题 |
| 腾讯 | 基础不够，尚未开始 |

### 适用场景

L3适合以下类型的需求：

1. **标准化程度高**：有明确规范和模板
2. **复杂度较低**：不涉及复杂架构设计
3. **风险可控**：失败可快速回滚
4. **重复性强**：类似需求多次出现

例如：
- 配置页面生成
- 标准CRUD接口开发
- 简单Bug修复
- 文档自动生成

### 实施建议

1. **从低风险场景开始**：选择失败成本低的场景试点
2. **建立安全网**：完善的监控、回滚机制
3. **明确边界**：清晰定义AI能力和人工介入的边界
4. **持续学习**：AI从人工修正中学习，不断提升能力

## L1→L2→L3 演进路径

### 演进原则

```
L1 AI辅助 (6-12个月)
    ↓
    ├── 工具选型与集成
    ├── 开发者培训
    ├── 基线度量建立
    └── 目标：采纳率>60%
    ↓
L2 AI协同 (12-18个月)
    ↓
    ├── 试点团队选择
    ├── 平台深度集成
    ├── 方法论沉淀
    └── 目标：L2需求占比>20%
    ↓
L3 AI自主 (18-24个月+)
    ↓
    ├── 低风险场景试点
    ├── 安全网建设
    ├── 边界定义
    └── 目标：L3需求占比>10%
```

### 关键成功因素

| 因素 | L1 | L2 | L3 |
|------|----|----|----|
| **主导团队** | 工具团队 | 效能团队 | 效能+AI团队 |
| **核心度量** | 采纳率 | 交付周期 | 交付成功率 |
| **平台要求** | IDE集成 | 全流程集成 | 端到端能力 |
| **组织准备** | 培训 | 流程标准化 | 高度标准化 |

## 五阶段模型与L1/L2/L3的映射

```
阶段1: 在线化 & 平台化 → 统一研发平台，工具渗透率>90%
    ↓
阶段2: 标准化 → 流程规范化，自动化>85%
    ↓
阶段3: 数字化 & 精益化 → 效能度量体系化
    ↓
阶段4: 智能化1.0 → L1 AI辅助（个人效率提升）
    ↓
阶段5: 智能化2.0 → L2/L3 AI协同/自主（组织效能提升）
```

### 各阶段前置条件

| 目标阶段 | 前置条件 |
|----------|----------|
| L1 | 完成阶段1-2（平台化+标准化） |
| L2 | 完成阶段3（数字化），L1采纳率>60% |
| L3 | L2需求占比>20%，流程高度标准化 |

## 实施路线图（2026年版）

### Phase 1：L1普及（2026 Q1-Q2）

**目标**：L1采纳率>60%

**关键动作**：
1. 统一AI编程工具选型
2. IDE深度集成
3. 全员Prompt工程培训
4. 建立L1度量体系

### Phase 2：L2试点（2026 Q2-Q3）

**目标**：2-3个标杆团队，L2需求占比>20%

**关键动作**：
1. 选择试点团队（流程规范、技术栈统一）
2. AI工具与平台深度集成
3. 定义L2开发方法论
4. 建立L2度量体系（交付周期为核心）

### Phase 3：L2规模化（2026 Q3-Q4）

**目标**：L2需求占比>30%

**关键动作**：
1. 总结试点经验，形成最佳实践
2. 培训推广L2开发方法
3. 扩大试点范围
4. 建立L2效能度量仪表板

### Phase 4：L3探索（2027+）

**目标**：低风险场景试点

**关键动作**：
1. 识别适合L3的场景
2. 建设端到端Agent能力
3. 建立安全网（监控+回滚）
4. 探索人机协作新范式

## 常见陷阱与规避

### 陷阱1：跳过L1直接做L2/L3

**现象**：没有打好个人效率基础，直接追求组织效能

**后果**：开发者不熟悉AI协作，L2/L3难以落地

**规避**：先完成L1普及，再推进L2

### 陷阱2：AI团队主导而非效能团队

**现象**：AI技术团队主导落地，缺乏组织视角

**后果**：技术与业务割裂，难以产生实际效能

**规避**：由效能团队主导，AI团队提供技术支撑

### 陷阱3：度量指标错误

**现象**：以代码生成率、采纳率为核心指标

**后果**：忽视了真正的组织效能（需求交付周期）

**规避**：以需求交付周期为核心北极星指标

### 陷阱4：平台与AI割裂

**现象**：AI工具与研发平台各自独立

**后果**：开发者体验割裂，效率不升反降

**规避**：AI能力与研发平台深度集成
