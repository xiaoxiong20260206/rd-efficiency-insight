# 字节跳动研发效能提升之道：标准化 → 数字化 → 智能化

> 来源：单虓晗 2025年QECon大会演讲
> 整理时间：2026-02-25

## 📌 核心框架

### 研发效能三层次

| 层次 | 关注角色 | 核心关注 | 目标 |
|------|----------|----------|------|
| **业务层** | 业务负责人 | 业务价值流动 | 业务目标达成、业务成功 |
| **研发交付层** | 技术Leader/PM/QA | 需求流动效率 | 需求持续、快速、高质量交付 |
| **技术实现层** | 一线开发人员 | 研发活动效率 | 顺畅、高效、低成本完成 |

### 效能提升三阶段

```
在线化 → 标准化 → 数字化
   ↓         ↓         ↓
效能平台  效能实践  效能度量
```

1. **在线化**：把线下研发过程搬到线上（效能平台）
2. **标准化**：沉淀研发标准和最佳实践
3. **数字化**：量化研发过程，构建指标体系，驱动改进

### 研发效能黄金三角（张乐）

```
      效能平台
         △
        / \
       /   \
      /     \
  效能实践 ◄─► 效能度量
```

三者构成**持续迭代的增强回路**：
- 效能实践 → 沉淀到效能平台
- 效能平台 → 承接实践落地和规模化
- 效能度量 → 洞察数据，找到瓶颈，优化实践和平台

## 🏗️ 字节跳动实践案例

### 建设效能基础设施

#### 1. 效能平台：Bits 研发平台

**发展历程**：
- **2019年前**：点 - 单点工具建设，"群雄割据"，90+款工具
- **2019-2020**：线 - 向三端整合（服务端、客户端、前端）
- **2022年起**：面 - 统一平台Bits形成

**建设结果衡量指标**：
- 目标用户覆盖率
- 用户量（日活/周活）
- 用户满意度

#### 2. 效能实践：研发最佳实践体系

**演进过程**：
- **2019-2022**：各业务线优秀实践涌现，按季度征集促进学习
- **2023**：提炼共性实践，形成"研发最佳实践"
- **2024至今**：抽象标准研发模式，全公司推广

**入选实践的3个标准**：
1. 具有普适性且被验证有效
2. 明确的目标和度量指标
3. 有公司级平台可支持

**核心实践（约20个）**：
每个实践包含：背景、目标与指标、实践方案、成功案例

#### 3. 效能度量：DevMind体系

**四阶段演进**：
| 阶段 | 时间 | 核心内容 |
|------|------|----------|
| **数据** | 2020-2021 | 构建数据中台，连接各工具数据 |
| **度量** | 2021 | 引入标准指标体系，形成组织级指标白皮书 |
| **洞察** | 2022 | 自动洞察报告，降低数据分析门槛 |
| **管理** | 2023至今 | 研发管理驾驶舱，"一张图"管理 |

**评估维度**：
- 数据：接入工具数量、有效数据资产
- 指标：有价值的权威指标数量
- 产品：用户量和满意度

## 🎯 分层分级提效

### ① 公司级驱动

**核心抓手**：《研发效能分析报告》
- 每年发布两次
- 包含各项研效指标基准数据
- 分析半年来的变化和原因

**作用**：
- 为公司管理层提供有价值输入
- 让各业务线了解公司基线，看清自己的差距

### ② 业务线级提效（100-1000人）

**0→1阶段：导入最佳实践**

5个实践逐步实施：
1. **研发流程标准化**：标准流程串联开发到发布
2. **研发质量防控**：每个环节加入质量防控
3. **研发&需求双流联动**：自动同步需求状态
4. **研发全流程洞察与改进**：基于数据分析改进
5. **一站式研发体验**：持续关注体验和效率

**1→100阶段：持续洞察与改进**

类比"体检"：量化现状 → 诊断问题 → 驱动解决

**千人级团队案例（2B业务）**：
- 背景：1000+人，季度交付4600+需求，平均交付周期44天
- 问题分析：微观采样 + 宏观指标印证
- 四类问题：需求复杂度、资源流程、研发质量、研发效率
- 效果：需求交付周期显著下降

### ③ 团队级改进（<50人）

**诊断流程**：
1. **业务层诊断**：业务目标和达成度
2. **交付效率诊断**：需求交付指标，找核心问题
3. **交付质量诊断**：线上缺陷分析，找改进措施
4. **研发基本功诊断**：按研发阶段找细节问题

**结果指标**：
- 需求平均交付周期
- 需求交付数
- 万行责任事故数
- 线上严重缺陷逃逸率

### ④ 个人级优化

**影响效率的两类因素**：
| 环节类型 | 影响点 | 优化目标 |
|----------|--------|----------|
| 自动化环节（编译/部署） | 反馈时间 | 缩短每个环节的反馈时长 |
| 人工环节（编码/调试/评审） | 返工浪费 | 一次性把事做对，消除返工 |

## 💡 原理与思考

### 软件工程的本质

> 软件研发不是工厂流水线，而是**复杂手工业**——"知识手工业者"的大规模协作。

**困境**：万人企业 ≈ 2000+最小交付团队，整体提升几乎"不可能"

**破局思路**：负反馈调节控制
1. 假定公司是复杂系统（黑箱）
2. 设置控制器，持续注入能量
3. 观察反馈，对比与目标的差距
4. 再次调节，循环往复

### 效能提升工作"三眼"心法

| 视角 | 含义 | 应用 |
|------|------|------|
| **鸟眼** | 全局视角 | 掌握通用解决方案，了解理想状态，找到与现实的落差 |
| **鱼眼** | 趋势视角 | 把握行业发展阶段、组织导向、技术演进，顺势而为 |
| **虫眼** | 细节视角 | 贴近地面观察，分析具体Case，"趴着"解决问题 |

### 关于"圆轮子"与"方轮子"

两个思考：
1. **基建建设角度**：首先保证自己的"轮子"确实是"圆"的
2. **基建应用角度**：如果轮子"圆"但卖不出去

## 🚀 AI时代的新挑战：三线分裂问题

> 基于《AI × 研发效能 2026洞察报告》补充

### 背景

AI技术的爆发给字节跳动的研发效能体系带来了新的挑战。原本整合好的研发工具链，在AI来临后再次分裂为三条独立的产品线。

### 三线产品矩阵

| 产品 | 定位 | 状态 | 核心问题 |
|------|------|------|----------|
| **Trae** | AI编程IDE | 只对外商业化 | 未与内部Bits平台打通，轻于服务内部 |
| **Bits-AI** | 平台单点AI能力 | 功能升级 | 只是功能点增强，无代差级变化 |
| **Aime** | 企业数字员工/AI助手 | 独立产品形态 | 未与研发工具链打通 |

### Aime 详解

**Aime** 是字节跳动内部开发的**多智能体助手**（AI Multi-agent），属于企业内部效率工具。

#### 核心数据

| 指标 | 数据 |
|------|------|
| **复杂问题解决率** | 77.6%（GAIA基准测试） |
| **代码任务完成率** | 66.4% |
| **网页操作成功率** | 92.3% |
| **效率提升** | 10倍 |

#### 产品定位

- **内部效率工具**：与 TRAE、飞书知识库、DataAgent 并列
- **企业数字员工**：面向内部员工的AI助手
- **多智能体架构**：支持复杂任务的分解与执行

#### 与字节AI产品矩阵的关系

```
字节跳动AI产品矩阵
├── 📱 2C消费端：豆包、扣子、即梦AI、猫箱
├── 🏢 2B企业服务：火山引擎、扣子企业版、飞书AI、HiAgent
└── 🔧 内部效率：Aime、TRAE、飞书知识库、DataAgent
```

### 三线分裂的根本问题

```
原本整合好的研发工具链（Bits平台）
        ↓
    AI来临后
        ↓
┌─────────────────────────────────────────────────────────────┐
│                    再次分裂为三条线                           │
├─────────────────┬─────────────────┬───────────────────────────┤
│     Trae        │    Bits-AI      │        Aime              │
│   AI编程IDE     │  平台单点能力    │     企业数字员工          │
├─────────────────┼─────────────────┼───────────────────────────┤
│ 只对外商业化    │ 只是功能升级     │    独立产品形态           │
│ 未与内部打通    │ 无代差级变化     │    未与工具链打通         │
└─────────────────┴─────────────────┴───────────────────────────┘
                         ↓
          三者都没有彻底解决：
          - 工程师的个人开发效率提升
          - 组织的交付效率提升
```

### 现状数据

- **内部AI代码采纳率**：仅约20%
- **对比**：快手Kwaipilot与平台深度集成，组织效能显著

### 根因分析

1. **传统工具链与AI工具链断裂**
   - Trae聚焦商业化，与Bits平台割裂
   - Bits-AI只做功能点，不是系统性变革

2. **三个产品各自为战**
   - 没有统一的效能目标
   - 没有统一的度量体系
   - 没有统一的用户体验

3. **组织层面问题**
   - AI团队与效能团队平行结构，利益不一致
   - Trae作为独立业务线运营，只有对外商业化目标

### 改进建议

基于2026洞察报告的建议：

1. **整合三条产品线**，由效能团队统一主导
2. **让Trae承担对内服务目标**，与Bits平台深度集成
3. **将Bits-AI从功能升级提升到代差级变化**，系统性重构AI能力
4. **Aime与研发工具链打通**，而非独立产品形态
5. **统一目标**：将内部AI采纳率从20%提升到60%+

### 关键洞察

> **"先平台化再智能化"是系统论必然** — 字节在平台化阶段做得最好（Bits整合90+工具），但在AI时代需要重新审视平台与AI的融合方式。
>
> **AI不是更好的工具，AI是新的参与者** — 需要建立全新的协作范式，而不是在旧结构上叠加AI。

## 📝 后记（作者感悟）

> 自从2016年"入局"研发效能工作，从团队效能治理到工具、度量体系建设，再到公司级的效能提升，AI在效能领域的应用也早在2019年"专有模型时代"就开始尝试。

**两个兴趣点**：
1. **更大范围的改变现实**：跨越理论和实践之间的沟壑
2. **探索领域的边界和上限**：从理论与现实的冲突中找到机会

**AI时代的新思考**：
- 研发效能的本质是**组织持续快速交付有效价值的能力**
- AI不会改变这个本质，但会**暴露和放大**组织的长板和短板
- 未来的研发效能工作，需要在**"人-AI协同"**的新范式下重新思考
