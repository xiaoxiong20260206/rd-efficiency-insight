## 📌 从零开始设计80分的Prompt

> 作者：张乐  
> 系列：3篇  
> 发布时间：2024年6月

### 核心方法：四步法

```
┌─────────────────────────────────────────┐
│           Prompt设计四步法              │
├─────────────────────────────────────────┤
│  1. 设定角色 - 赋予模型"能力包"        │
├─────────────────────────────────────────┤
│  2. 描述任务 - 明确目标，定义任务       │
├─────────────────────────────────────────┤
│  3. 拆分动作 - 描述具体行动或步骤       │
├─────────────────────────────────────────┤
│  4. 明确要求 - 描述输出结果的期望       │
└─────────────────────────────────────────┘
```

### 第一步：设定角色

**核心原则**：角色定义越具体越好

**示例**：
```
你现在是一名资深代码审查员，具有超过15年的软件开发经验，
精通Python和代码优化。你熟悉常见的编程错误、性能优化技巧
以及代码可读性最佳实践，可以在审查代码的过程中找到潜在问题，
并提供改进建议。
```

### 第二步：描述任务

**核心原则**：任务描述应尽量具体和详细

**示例**：
```
你现在是一名资深软件工程师，具有丰富的代码重构和性能优化经验，
精通Python编程，熟悉设计模式和代码最佳实践。请帮助我重构以下
Python代码，使其更具可读性、可维护性，并提高性能。代码如下：xxxxx。
```

### 第三步：拆分动作

**核心原则**：将复杂任务拆分为多个具体步骤

**示例（测试用例设计）**：
1. 理解需求：深入理解软件需求和功能
2. 编写测试点：确定需要测试的功能和特性
3. 编写测试用例：对每个测试点编写详细用例
4. 评审测试用例：确保完整、准确、易于理解

### 第四步：明确要求

**常见格式要求**：
- 清晰的逻辑结构：使用段落和标题
- 项目符号和序号：增加可读性
- 代码块和引号：区分强调内容
- 关键词强调：加粗或斜体
- 表格和矩阵：展示数据或对比信息

### 实战案例：运维告警摘要

```
你是一位运维专家，我将给出若干条来自监控系统的告警消息，
请按照以下要求生成这些告警消息的总结摘要；

【输出要求】
1. 分析这些告警的发生时间是否存在规律
2. 如果告警内容包含告警等级，则需要给到各个告警等级的占比
3. 对不同告警中存在的相同内容，进行简要的描述
4. 从运维专家角度，分析可能导致告警的根本原因并用一句话描述
5. 结合告警内容，给出三条可行的处理建议
6. 除了以上内容，不要输出其他内容
7. 如果不能给出回答，请按原样回复输入内容，不要编造回答

【格式要求】
# 告警摘要
**时间规律**：xxx；
**告警等级**：xxx；
**告警内容**：xxx；
**可能原因**：xxx；
**处理建议**：
**1**. xxx；
**2**. xxx;
**3**. xxx
```

## 📌 CI流水线报错智能分析

> 作者：张乐  
> 系列：6篇  
> 发布时间：2024年7月

### 一、背景与目标

**痛点**：
- 用户需通过聊天描述问题、截图、链接发给客服
- 客服人力有限，需排队处理
- 需重复执行信息查找和日志分析

**目标**：
- 在第一现场定位问题
- 给出问题原因和解决方案
- 通过操作链接或工具调用一键解决问题

### 二、技术方案

#### 整体架构

```
用户查询 → 初步分析 → 深度分析 → 问题诊断 → 解决方案推荐
               │            │
               ↓            ↓
          私域知识库      日志分析
```

#### 初步分析
- 结合私域知识库进行检索增强（RAG）
- 对大模型进行针对性精调训练

#### 深度分析
- 获取插件报错详细日志
- 结合大模型的世界知识给出诊断

### 三、核心技术挑战

#### 挑战1：日志超长问题

**问题**：
- 报错日志经常数百MB，极端可达GB级
- 上下文越多，大模型推理准确度越低

**解决方案对比**：

| 方案 | 效果 |
|------|------|
| 直接截取尾部N行 | 一般 |
| 正则匹配特定关键词 | 较好 |
| **正则匹配关键词+临近N行** | **最佳** |
| 主流日志异常识别算法 | 效果不稳定 |

**关键词获取**：
- 让大模型列举常见编程语言、编译工具的错误关键词
- 后续通过bad case不断增补

#### 挑战2：关联日志提取

**方法1：基于图的关联日志提取算法**

```
日志预处理 → 节点和边定义 → 构建有向图
    ↓
报错日志提取 → 关联token挖掘(N跳) → 关联日志挖掘 → 大模型分析
```

**方法2：利用大模型提取关联日志**
- 增加一次大模型调用
- 准确性更高，但耗时更长

### 四、效果评价体系

#### 6个评估指标

| 指标 | 评估内容 | 实现方式 |
|------|----------|----------|
| **AnalysisScorer** | 是否捕获关键错误信息 | 关键词提取+语义相似度 |
| **SolutionScorer** | 解决方案是否匹配 | 关键词提取+语义对比 |
| **FormatScorer** | 输出格式是否符合要求 | 规则检查 |
| **SpecificityScorer** | 是否足够具体（非宽泛） | TF-IDF词频统计 |
| **AccuracyScorer** | 分析和方案是否准确 | SOTA大模型评分 |
| **CompletenessScorer** | 内容是否完整 | SOTA大模型评分 |

#### 数据飞轮机制

```
用户反馈(点赞/点踩) → Bad Case分析 → 针对性优化 → 追加评测集 → 持续提升
```

### 五、落地实现细节

- 日志处理效率优化
- OOM问题避免
- 模型推理速度提升（首字节+完整内容）
- 缓存策略
- 流式输出和内容渲染
- 人工兜底流程（一键拉群）

### 六、核心经验

> **最终线上使用的方案，是在性能和效果之间取得适当、相对的平衡。**

## 💡 Prompt设计Checklist

- [ ] 是否设定了具体的角色（经验、技能、专业领域）？
- [ ] 任务描述是否具体明确？
- [ ] 复杂任务是否拆分为具体步骤？
- [ ] 是否明确了输出格式要求？
- [ ] 是否提供了足够的背景信息？
- [ ] 是否针对不同用户群体设计了差异化策略？
- [ ] 是否有兜底处理（无法回答时的行为）？
