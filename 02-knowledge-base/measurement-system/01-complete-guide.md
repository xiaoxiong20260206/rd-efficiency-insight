## 📊 实践1-数据：建设研发数据中台

### 目标
解决数据分散、采集难、不准确、流程不规范等问题

### 建设思路

**思路1：开放数据**
- 开放研发数据给少部分懂数据的人（QA、PMO）
- 依靠"群众的力量"打磨数据

**思路2：封装服务**
- 不开放研发数据
- 基于宽表提供各类数据分析报表、大盘

### 核心方法：OneData数据建模

```
指标设计 → 用OneData方法梳理业务板块
    ↓
指标拆解 → 实现指标到数据模型的映射
    ↓
数据建模 → OLAP架构，星型模型
```

### 建设方案

```
源数据表 → 数据接入 → 研发数仓(DWD表) → 数据集 → 图表 → 仪表盘
```

### 经验总结

| 优点 | 难点 |
|------|------|
| 数据"越用越准" | 建设成本投入大 |
| 可快速淬炼高质量数据 | 收益不如工具容易被看清 |

> **建议**：以"最多干一年"的心态推进

## 📊 实践2-度量：建设研发指标体系

### 目标
沉淀专家经验，建设专业、统一、标准、可工程化实现的指标

### 两种方法结合

**张乐方法**：基于研发过程模型结合度量目标和经验搭建指标

**任晶磊方法（GQM）**：
- G（Goal）：目标
- Q（Question）：问题
- M（Metric）：指标

### 指标设计流程：研发指标流水线

```
体系建模 → 指标设计 → 指标实现
    ↓          ↓          ↓
  清晰定义   问题定义   数据研发
  问题域     量化分析   数据科学
  评价角度   指标定义   数据可视化
```

### 指标实现：研发指标即代码

- 取消数据应用层（ADS）
- 所有指标配置后直接SQL化
- 减少数据加工等待
- 默认支持明细下钻

### 关键指标度量经验

#### 研发产出度量

| 方式 | 特点 | 推荐度 |
|------|------|--------|
| 人天 | 主观性强 | 低 |
| 功能点 | 需要专业度量 | 中 |
| **代码行数** | 需定义精度标准 | **高** |

#### 需求交付周期度量

| 要点 | 说明 |
|------|------|
| 数据范围 | 只统计"标准流程"需求 |
| 统计方式 | 分位数或平均数均可 |
| 推荐 | 使用平均数（更灵活的改进策略） |

## 📊 实践4-管理：研发管理驾驶舱

### 目标
固化管理实践，赋能所有业务、团队在线化分析、决策、驱动改进

### 核心理念：MARI循环

```
M:度量 → A:分析 → R:回顾 → I:改进 → M:度量 ...
                     ↑
              持续推动改进、螺旋式上升
```

### 业产研数字化管理

| 环节 | 内容 |
|------|------|
| **业务设计** | 明确团队目标并量化监控，清晰划分业务方向 |
| **业务执行** | 监控目标达成 |
| **研发交付** | 体检工程情况 |
| **人员管理** | 评估人员研发情况 |

### 管理地图核心能力

- 为技术Leader提供统一视图
- 多视角管理（业务/项目/团队/系统）
- 目标进度监控

### 应用案例：C业务线

**搭建步骤**：
1. 产品方向分解：明确年度战略，划分业务方向
2. 目标绑定指标：组织→团队→个人目标互锁
3. 形成驾驶舱

**改进效果**：
| 指标 | 改进前 | 改进后 |
|------|--------|--------|
| 事故监控召回率 | 25% | 50% |
| 事故MTTR达标率 | 0% | 50% |
| 头部渠道有效管控率 | 7.1% | 50.6% |
| 老功能召回率 | 58.1% | 78.6% |
| 缺陷逃逸率 | 29.41% | 18.24% |

## 💡 核心洞察

### 度量体系的"三个关键"

1. **数据是基础**：没有准确的数据，一切度量都是空中楼阁
2. **指标是桥梁**：连接数据与业务目标
3. **洞察是价值**：让数据真正驱动改进

### 推广的优先级

```
业务线级（最核心，承上启下）
    ↑
公司级（全局效能报告）
    ↓
团队级（标准化效能体检）
    ↓
个人级（通过工具提效，不做个人度量）
```

### 体系建设的"三年周期"

```
第一年：做工具（在线化）
    ↓
第二年：推实践（标准化）
    ↓
第三年：做度量（数字化）
    ↓
规划"下一代工具链"才能再来一个三年
```
